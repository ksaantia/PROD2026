1. Резюме (Executive Summary)
   Данный документ описывает целевую архитектуру системы Wellness Gift Shop. Проект реализует паттерн Modular Monolith, сочетающий высокую скорость разработки с масштабируемостью бизнес-логики. Основной упор сделан на Data-Driven архитектуру (фильтрация на стороне БД) и Security-first approach (отказ от публичного доступа к медиа-ресурсам).

2. Архитектурная декомпозиция и потоки данных

2.1. Логическое разделение слоев
Система разделена на 4 изолированных слоя для упрощения тестирования:

Transport Layer (Gin/HTTP): Отвечает за сериализацию, десериализацию и первичную валидацию (DTO).

Service Layer (Order/Product Services): Слой бизнес-логики. Здесь происходит оркестрация (например, связь OrderRepo и ProductRepo).

Data Access Layer (GORM Repositories): Инкапсулирует SQL-запросы.

Storage Layer (MinIO SDK): Управление жизненным циклом медиа-файлов.

3. Инженерные компромиссы (Trade-offs)

3.1. S3-хранилище: безопасность, латентность
Альтернатива: публичный бакет.

Отказ: Риск сканирования контента (directory listing).

Выбранный путь: Presigned URLs.

Пояснение: мы жертвуем дополнительным сетевым запросом к MinIO (для получения URL), чтобы гарантировать, что контент доступен только авторизованному клиенту в течение короткого окна. Это решение полностью защищает нас от использования нашего контента на сторонних ресурсах.

3.2. Хранение цен: Decimal, Float
Обоснование: Использование numeric(12, 2) в PostgreSQL вместо float64 — это промышленный стандарт для финтех-операций. Это исключает классическую ошибку округления при сложении (например, 0.1 + 0.2 != 0.3).

3.3. API: синхронность, асинхронность
Отправка email реализована через Goroutines.

Обоснование: Почтовые серверы часто имеют высокий TTI (Time to Initial). Мы не заставляем клиента ждать, пока Google ответит на наш SMTP-запрос. Заказ считается принятым сразу после записи в Postgres.

4. Оптимизация производительности

4.1. Полнотекстовый поиск (Search Engine Optimization)
Для поисковой строки используется Postgres Full-Text Search:

code
SQL
CREATE INDEX idx_gifts_search ON gifts USING gin(to_tsvector('russian', name || ' ' || description));
Это переносит нагрузку с приложения на БД. При поиске мы вызываем to_tsquery, что обеспечивает мгновенный ответ даже при 100 000+ товаров.

4.2. Сетевой трафик
Мы используем "Payload Slimming":
Фронтенд запрашивает только то, что нужно. В объекте Gift нет избыточных полей. При получении списка товаров ImageURL подгружается только для тех объектов, которые реально отображаются на экране (Lazy Loading подход).

5. Стратегия безопасности (Security Policies)
   Input Sanitization: Использование DTO-объектов models.go гарантирует, что в систему попадают только те данные, которые мы ожидаем. Весь лишний "мусор" отсекается на уровне binding.

Infrastructure Security: вся конфигурация хранится в .env, доступ к БД закрыт извне. MinIO требует Access/Secret ключей для любой операции.

Error Handling Policy: система возвращает клиенту только санитизированные ошибки. Внутренние ошибки БД или инфраструктуры логируются на сервере, но не отдаются клиенту (во избежание утечки информации об архитектуре системы через стектрейсы).

6. План развития (Future Roadmap)
   Caching Layer: внедрение Redis для кэширования результатов GetProducts на 5-10 минут (снизит нагрузку на БД при частых запросах к популярным категориям).

Message Broker: замена Goroutine на RabbitMQ для отправки Email, если потребуется надежность доставки 99.99%.

Audit Log: внедрение таблицы для аудита всех действий пользователей (кто, когда и какой подарок купил).
