RFC-001: Техническая архитектура и протоколы взаимодействия платформы Wellness Gift Shop
Дата: 17 мая 2026 г. Авторы: Волейко Матвей, Полупанова Ксения, Веренич Егор, Касперович Иван. 10 команда

1. Обзор
   Данный документ описывает техническую архитектуру платформы Wellness Gift Shop. Система спроектирована как высокопроизводительное SPA-приложение с фокусом на безопасность передачи данных, оптимизацию ресурсов хранилища и надежность API-контрактов.

2. Безопасность и архитектура хранения

2.1. Защита медиа-ресурсов (S3 Architecture)
Для обеспечения контроля над медиа-контентом реализована модель «закрытого хранилища»:

Изоляция доступа: Все исходные медиа-файлы хранятся в S3-бакете, доступ к которому извне (public access) полностью запрещен.

Абстракция: База данных хранит только уникальные идентификаторы (ImageKey). Внешние системы не имеют доступа к структуре путей хранилища.

Динамическая доставка (Presigned URLs): Клиентскому приложению передаются Presigned URL — временные авторизованные адреса, генерируемые сервером «на лету».

Защита от компрометации: Сгенерированная ссылка имеет строго ограниченный TTL (15 минут). После истечения этого времени ссылка становится невалидной, что делает невозможным прямой доступ к ресурсам S3.

2.2. Безопасность на прикладном уровне (HTTP)
Взаимодействие между клиентом и сервером осуществляется по протоколу HTTP с усиленной валидацией на уровне бизнес-логики:

Валидация запросов: Каждый входящий запрос проходит строгую схематичную проверку (DTO validation) на бэкенде. Система отсекает некорректные пакеты данных до того, как они попадут в логику обработки заказа.

Изоляция параметров: Все идентификаторы ресурсов проверяются на стороне бэкенда. Любые попытки манипуляции параметрами API блокируются на уровне бизнес-логики, предотвращая несанкционированный доступ к данным.

Целостность данных: Поскольку система не передает клиенту прямые пути к файлам, риск подмены или перехвата ресурсов минимизирован самим архитектурным подходом к генерации Presigned URL.

3. Производительность API

3.1. Backend-Driven Filtering
Вся вычислительная нагрузка по поиску, фильтрации и сортировке перенесена на сторону СУБД (PostgreSQL). Фронтенд получает только готовый JSON-ответ, что исключает «тяжелые» вычисления на стороне браузера и обеспечивает отклик системы < 1.5 сек.

3.2. Контракт API (OpenAPI 3.0)
Использование Swagger (OpenAPI 3.0) обеспечивает:

Строгую типизацию: Фиксация схем (Gift, OrderRequest).

Авто-валидацию: Сервер автоматически отвергает запросы, не соответствующие спецификации, что защищает систему от инъекций и ошибок интеграции.

4. Инфраструктурная надежность
   CI/CD: Автоматизированные конвейеры проверки кода (linting/tests) и деплой через Docker-контейнеры.

Мониторинг: Сквозное JSON-логирование запросов обеспечивает прозрачность всех операций для оперативного аудита и отладки.
